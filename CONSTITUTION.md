# CONSTITUTION.md — Принципы проекта NetCertGuardian

## Назначение документа

Этот документ задаёт неизменяемые принципы разработки. Любой участник — человек или ИИ-агент — обязан следовать им. Принципы имеют приоритет над краткосрочными соображениями скорости.

---

## 1. Корректность важнее скорости

- Программа **не должна падать** при недоступности части хостов. Частичный результат лучше краша.
- Все ошибки сетевого взаимодействия **логируются, не замалчиваются**.
- Если метод сбора данных не сработал — хост попадает в `errors`, а не игнорируется.

## 2. Честная модель покрытия

- Программа **не утверждает**, что проверила 100% сети, если часть хостов была недоступна.
- В отчёте явно указаны: всего обнаружено / успешно опрошено / ошибок.
- Администратор видит реальную картину, а не оптимистичную.

## 3. Один запуск — один цикл

- Программа запускается, делает свою работу, завершается.
- **Нет демонов, нет фоновых процессов, нет sleep-петель** в рабочем коде.
- Регулярность обеспечивается планировщиком Windows (Task Scheduler), не программой.

## 4. Zero pre-configuration на целевых машинах

- Программа не требует установки агентов, включения служб или изменения GPO на опрашиваемых компьютерах.
- Единственное предположение: работающий SMB (порт 445) и права локального администратора.
- Если машина не отвечает — это ошибка, а не повод менять её конфигурацию.

## 5. Self-contained отчёт

- `cert-status.html` — **единственный файл**, который нужно открыть администратору.
- HTML работает без интернета, без сервера, без дополнительных файлов.
- Все данные вшиты в HTML как JavaScript-переменная при генерации.

## 6. Ротация без исключений

- Программа **никогда** не оставляет неограниченное количество файлов в `reports/`.
- После каждого запуска выполняется ротация согласно конфигу.
- Нет конфига ротации — используются безопасные дефолты (30 запусков, 90 дней).

## 7. Безопасность учётных данных

- Пароль **никогда** не попадает в логи, CSV, HTML или JSON-отчёты.
- Пароль можно хранить в `config.yaml` в открытом виде (по требованию заказчика), но файл создаётся с ограниченными правами.
- В будущем: поддержка Windows Credential Manager (опционально).

## 8. Типизация и качество кода

- Весь Python-код использует **type hints** (PEP 484).
- Dataclass-модели для передачи данных между модулями.
- Pydantic для валидации конфигурации.
- Стандартный `logging` (не print) для всех диагностических сообщений.

## 9. Тестируемость

- Сетевые операции изолированы в отдельных модулях (`discovery.py`, `collector.py`).
- Бизнес-логика (`analyzer.py`, `rotation.py`, `reports.py`) тестируется без сети.
- Минимальное покрытие тестами: конфиг, анализ, ротация, генерация отчётов.

## 10. Явное лучше неявного

- Конфиг отсутствует → создаём шаблон → сообщаем → `exit(0)`. Не угадываем.
- Обязательные параметры не заполнены → сообщаем что именно → `exit(1)`.
- Непредвиденная ошибка → логируем traceback → `exit(1)`.

---

## Exit codes

| Код | Значение |
|-----|----------|
| 0   | Успешное завершение (в т.ч. создание конфига при первом запуске) |
| 1   | Критическая ошибка (нет сети, невалидный конфиг, системная ошибка) |
| 2   | Частичный успех — не использовать (все ошибки логируются, exit=0) |
